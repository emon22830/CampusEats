## What This Is
Campus food pre-order app for UTM food courts. Customers browse vendors, order
ahead, and skip the pickup queue. Vendors manage their menu/orders; admins
approve vendor signups.

## Structure
- `CampusEats-frontend-dev/frontend/` — Vue 3 + Vite SPA (Pinia, Vue Router)
- `CampusEats-backend-dev/backend/` — PHP 8.1+ Slim API (JWT auth, MySQL via Docker Compose)
- `CampusEats-mobile-dev/` — Capacitor shell (iOS/Android native projects) wrapping the frontend's build output

## Backend (local dev)
```
cd CampusEats-backend-dev/backend
docker-compose up -d        # mysql + phpmyadmin + backend on :8080
```
`docker compose` (the plugin form) isn't installed on this Mac — use the
standalone `docker-compose` binary (`/opt/homebrew/bin/docker-compose`, v5.1.4).

CORS is currently `CORS_ORIGIN: '*'` in `docker-compose.yml` to support both
the Vite dev server and the Capacitor mobile WebView origins. Tighten this
before any public deployment.

## Frontend (web)
```
cd CampusEats-frontend-dev/frontend
npm run dev       # http://localhost:5173, talks to http://localhost:8080
```

## Mobile (Capacitor) — added 2026-07-05, moved to its own top-level folder same day
Wraps the frontend's Vue SPA into native iOS/Android shells, living in its own
sibling folder (`CampusEats-mobile-dev/`) rather than inside `frontend/`, so
`android/` can be opened directly in Android Studio and `ios/` directly in
Xcode. App ID: `com.solvenexify.campuseats`.

- `capacitor.config.json` — `webDir` points across folders at
  `../CampusEats-frontend-dev/frontend/dist` (Capacitor resolves it relative
  to wherever `capacitor.config.json` itself lives, so this works fine).
  `server.cleartext: true` allows plain-HTTP API calls during dev — remove
  once the backend is served over HTTPS. `server.androidScheme: "http"` is
  also required (see gotcha below) — remove alongside `cleartext` at the same
  time.
- `package.json` here only holds the Capacitor JS packages
  (`@capacitor/core|cli|ios|android`) — needed locally because `cap sync`
  wires the native Android project to `../node_modules/@capacitor/android`
  by relative path (regenerated by `cap sync`, so don't hand-edit
  `android/capacitor.settings.gradle`).
- `.env.mobile` (in `CampusEats-frontend-dev/frontend/`) — `VITE_API_BASE_URL`
  pointed at the dev machine's LAN IP (currently `10.217.99.121:8080`) since a
  device/emulator can't reach `localhost` meaning itself, not the Mac.
  **Update this IP whenever it changes** (check with `ipconfig getifaddr en0`),
  and swap it for the real API URL once the backend is deployed.
- iOS `Info.plist` has an ATS exception (`NSAllowsArbitraryLoads`) for the same
  reason — also dev-only, remove before release.
- Safe-area CSS (`env(safe-area-inset-top/bottom)`) was added to `App.vue`'s
  header/bottom-nav (in the frontend project) — Capacitor's WebView draws
  edge-to-edge, unlike a browser, so without this the header sits under the
  iOS status bar.

Build/run (from `CampusEats-mobile-dev/`):
```
npm run sync           # builds frontend in mobile mode, then cap syncs
npm run open:ios       # opens Xcode
npm run open:android   # opens Android Studio
```
Re-run `npm run sync` any time frontend source changes, before opening/running
the native IDEs — Android Studio/Xcode just build whatever's already in
`android/`/`ios/`, they don't rebuild the web bundle themselves.

iOS uses Swift Package Manager (no CocoaPods needed) via a pinned GitHub
package, so it has no local node_modules path dependency. Android Gradle
requires JDK 21 — pinned per-project via `android/gradle.properties`
(`org.gradle.java.home`) rather than changing the machine-wide `JAVA_HOME`
(17, used by other tools — see `~/dev-flow/CLAUDE.md`).

Both platforms were build-verified and run-verified (iOS Simulator + Android
emulator) against the live local backend on 2026-07-05, both before and after
the move to `CampusEats-mobile-dev/`.

### Gotcha: Android mixed-content silently masked by the mock-data fallback
Android's WebView loads the app at `https://localhost` by default (Capacitor's
default `androidScheme`), and Chromium blocks "mixed content" — an HTTPS page
fetching a plain `http://` API — **regardless of** `usesCleartextTraffic`/
`server.cleartext`. Those two settings are unrelated: `cleartext` permits
cleartext sockets at the OS level, mixed-content blocking is a separate
same-page-security-context rule enforced by the WebView itself.

This went undetected during initial testing because `client.js`'s vendor/menu
fetches silently fall back to bundled mock JSON (`public/mocks/`) whenever the
backend is unreachable (by design, so the storefront still browses if the
backend's down) — so the vendor list rendered fine even though every real API
call was being blocked. It only surfaced when login was tried, since auth has
no mock fallback and threw "Could not reach the server." **Lesson: don't treat
a rendered vendor list as proof the mobile app reached the real backend —
check logcat/Safari Web Inspector network tab, or the ratings against the
seed data, since mock data looks convincingly real.**

Fix: `server.androidScheme: "http"` in `capacitor.config.json`, so the app's
own origin is `http://localhost` — matching scheme with the `http://` API, no
mixed content. Re-run `npm run sync` and rebuild after changing this.
